# Wild Card

**Describe a tool you wish you had, and a few seconds later it's on your home screen, working, yours to keep.**

Wild Card turns a plain-language description ("a tip splitter that remembers my
usual tip", "a workout interval timer") into a small, working tool — generated by
AI, validated automatically, and run as an "app" on your device. The user never
has to see code, but can always open it, read it, and edit it.

This repository is the **engineering implementation**. The complete product
specification — vision, requirements, architecture, store-compliance strategy,
business model — lives in [`dev/`](dev/) (the "app bible", Docs 00–09).

---

## The one architectural idea

**Separate generation from execution.**

| | Where | What |
|---|---|---|
| **Generation** | Server | Strong AI models, retries, and automated validation turn a prompt into a self-contained web bundle. Heavy. |
| **Execution** | Device | A finished tool is plain HTML/CSS/JS that runs inside a sandboxed WebView. No server at run time. Light. |

A "tool" is therefore just a **self-contained bundle** (`index.html` + optional
assets) plus a manifest. It runs in an isolated WebView and talks to the host
only through a small, fixed JavaScript surface called the **Runtime SDK** (`WC.*`).

This split is also what makes the product **App Store-compliant**: tools are
HTML/JS the user owns and can edit (the scripting/automation lane, Apple 2.5.2 +
4.7), no native code is generated, and device actions (share/export) are
performed by the host's own native chrome — never by generated code. See
[`dev/07_STORE_APPROVAL.txt`](dev/07_STORE_APPROVAL.txt).

---

## Launch strategy: web first

The v1 channel is a **web/PWA app**, for three reasons:

1. **No App Review on day one** — fastest path to a paying user.
2. **The web runtime _is_ the future iOS runtime.** A browser `<iframe sandbox>`
   with a strict CSP is the same isolation model as the on-device WebView, so the
   `WC.*` SDK and the sandbox host are built **once** and embedded everywhere.
3. **The validation harness and the device runtime are literally the same code**,
   eliminating a whole class of "passed validation but broke on device" bugs.

When demand is proven, the same runtime is wrapped in a React Native shell for
iOS/Android. Nothing is thrown away.

## Getting started

Requires **Node 20+**.

```bash
# 1. Install all workspaces
npm install

# 2. Build the runtime (compiles the browser bundle the host + validator load)
npm --workspace @wildcard/runtime run build:global

# 3. Run the runtime test suite (downloads Chromium on first run)
npx --workspace @wildcard/runtime playwright install chromium
npm --workspace @wildcard/runtime test

# 4. Run the server (generation) tests — no API key required
npm --workspace @wildcard/server test

# 5. Exercise the full generation pipeline offline (stub model + real validator)
npm --workspace @wildcard/server run dryrun

# 6. Run the generation server locally (needs a provider key — see below)
cp .env.example .env   # then edit .env
npm --workspace @wildcard/server run dev
```

### Generation provider

The server reads its key **server-side only** (REQ-SEC-001) and prefers
**OpenRouter** when `OPENROUTER_API_KEY` is set, otherwise it falls back to
`ANTHROPIC_API_KEY`. Set `WC_MODEL` to a slug your provider understands
(OpenRouter: `anthropic/claude-sonnet-4`; Anthropic: `claude-sonnet-4-6`).
Smoke-test a live generation with `npm --workspace @wildcard/server run livegen -- "a tip splitter"`.

Generate a tool against the local server:

```bash
curl -N -X POST http://localhost:8787/v1/generate \
  -H 'content-type: application/json' \
  -d '{"prompt":"a tip splitter that remembers my usual tip %"}'
```

## How a generation works (Tier-1)

```
prompt ──▶ model (cached system prefix) ──▶ strict <wc-app> output
                                                   │
                                          parse → Bundle (contract.ts)
                                                   │
                                   validate in headless Chromium (validate.ts)
                                                   │
                          pass? ── no ──▶ repair (≤3 turns, gated by validator)
                            │ yes
                            ▼
                  deliver bundle → run in WebView on device
```

The repair loop is **bounded** and gated by the validator's concrete pass/fail —
never the model's self-assessment. A tool is never delivered until it has run
clean (REQ-GEN-002). If it can't be made to work, the user gets an honest
"want a simpler version?" message, never a broken icon (REQ-GEN-008).

## Security model (summary)

- **Untrusted by default.** Every generated/imported tool is treated as
  potentially adversarial.
- **Isolation:** each tool runs in its own `<iframe sandbox="allow-scripts">` (a
  unique null origin) — no access to the parent, to cookies, or to another tool's
  storage (REQ-RUN-002).
- **No arbitrary network.** A strict CSP (`connect-src 'none'`) blocks all direct
  egress; the only network path is `WC.net.fetch(provider, …)`, routed through a
  fixed catalog of **server-proxied** data providers (REQ-RUN-005, REQ-SEC-001).
- **Secrets stay server-side.** No provider API key is ever in a client or a
  bundle.

See [`AGENTS.md`](AGENTS.md) for the invariants any contributor (human or AI) must
preserve.
