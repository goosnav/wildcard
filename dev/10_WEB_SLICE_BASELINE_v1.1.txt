10.1  WHY THIS OVERLAY EXISTS
--------------------------------------------------------------------------------

  The v1.0 bible (baselined 2026-06-03) was written for the full General
  Availability program: iOS + Android, Tier-1 and Tier-2 generation, RN shell,
  cross-device encrypted sync, and a 5-6 FTE team working 7-9 months (Doc 05).

  Since 2026-06-06 the repository has been building the LEAN REVENUE-FIRST
  SLICE the bible itself recommends in §5.6: a single web/PWA channel, Tier-1
  generation only, no second platform, no sync. That slice is the v1.1
  release; the rest of the v1.0 program is the v1.0+ roadmap. This overlay
  exists so:

    - A new contributor can read STATUS.md + this doc + the relevant bible
      section and not be confused by docs that describe a product larger than
      what is shipping.
    - A reviewer can verify that v1.1 is consistent with the bible's intent
      (tenets, security model, store posture) even where the scope is smaller.
    - When v1.0+ work begins (iOS, Android, Tier-2, sync) it is unambiguous
      which bible sections come back into scope.

  The v1.0 IDs (REQ-*, CMP-*, TC-*, M*, RSK-*, OQ-*) are STABLE and reused
  unchanged in v1.1. We annotate status; we do not renumber.

10.2  v1.0 vs v1.1 — SIDE-BY-SIDE SCOPE
--------------------------------------------------------------------------------

  v1.0 (the long-term GA target — Doc 05 §5.4, Doc 01 §7)
    - iOS app (React Native + thin native modules for WebKit/StoreKit).
    - Android app (React Native + Android-native bridge for richer capabilities).
    - Tier-1 generation (single model call + bounded repair loop).
    - Tier-2 agentic builds (Claude Agent SDK in an ephemeral cloud sandbox).
    - Live build preview, AI editing with version history, source view/edit.
    - Home grid with folders, drag, long-press actions, search.
    - Local-first storage with optional encrypted cross-device sync.
    - IAP via StoreKit (iOS) and Play Billing (Android), with RevenueCat.
    - Content manifest + privacy disclosures + Apple Privacy Label + Play
      Data Safety form.
    - Eval harness >=200 prompts gating prompt/model changes in CI.
    - External pentest, accessibility audit, store submission and review.

  v1.1 (the web slice being shipped first — this doc, Doc 05 §5.6)
    - Web/PWA app (React + Vite shell, single container serves API + UI).
    - Tier-1 generation only (no Tier-2 agentic builds).
    - Live build preview via SSE, "Keep it" / "Discard" flow, home grid.
    - Source view in READ-ONLY mode (edit deferred; see ROADMAP.md).
    - Magic-link email auth, server-enforced free-build quota, Stripe checkout
      for Pro subscriptions via web (not IAP — IAP is a v1.0+ concern).
    - JSON or Postgres storage; Resend magic links; Docker-deployable.
    - Server-proxied data providers (weather, currency) behind a fixed catalog
      and an allow-list (REQ-RUN-005).
    - Eval harness with the 27-prompt starter corpus (target gate: >=85%
      first-try, >=200 prompts at v1.0).
    - Posture and store-readiness for a WEB launch only. iOS / Android builds
      are deferred. The same Runtime SDK and bundle format ship in v1.1 and
      are wrapped unchanged in v1.0 (Doc 03 §3.1; Doc 07 §7.5).

  Items DELIBERATELY not in v1.0 either (per Doc 01 §7): multi-user sharing,
  a public gallery/marketplace, team/collaboration features, web embedding of
  others' tools, plug-in ecosystems. These are post-GA bets.

10.3  v1.1 ARCHITECTURE (re-states Doc 03 in v1.1 terms)
--------------------------------------------------------------------------------

  v1.1 IS the architecture described in Doc 03 §3.1-3.8, restricted to the
  web channel. The on-device runtime (Doc 03 §3.2) is the browser, not a
  native WebView. Everything else is identical.

  - GENERATION (CMP-03 + CMP-04 + CMP-05): server-side. Tier-1 only. The
    Model Gateway prefers OpenRouter (with 429 retry), falls back to Anthropic,
    and supports a deterministic stub for offline plumbing checks.
  - EXECUTION (CMP-02 + CMP-07s): browser-side, in an `<iframe sandbox>`
    with a strict CSP (no `allow-same-origin`). The injected `WC` global is
    built ONCE from `runtime/src/sdk.ts` and is the SAME artifact the
    server-side validator loads (CMP-05 parity; Doc 03 §3.8; AGENTS.md).
  - PERSISTENCE (CMP-09 / CMP-10 / CMP-11 partial): local-first in the
    browser (IndexedDB via `host-web/src/idb.ts`) + server (Postgres or JSON
    file via `server/src/store.ts`). Cross-device sync is v1.0+ (REQ-DATA-002).
  - BILLING (CMP-09 partial): Stripe Checkout for web subscriptions
    (`server/src/stripe.ts` + `server/src/billing.ts`). Entitlements enforced
    server-side (REQ-PAY-003, REQ-NFR-006).
  - MODERATION (CMP-12): NOT YET IMPLEMENTED in v1.1 (see STATUS.md matrix;
    v1.1 ships a Trust-and-Safety surface later — web launch tolerates a
    pre-screen/post-screen gap because there is no App Review gating it).
  - OBSERVABILITY (CMP-13 partial): `/health` reports subsystem state; an
    allow-listed admin dashboard exposes a roll-up view. Per-user rate
    limits, cost dashboards, and full observability are v1.0+.

  Two trust boundaries the v1.1 spec inherits UNCHANGED from Doc 03 §3.8:
    1. The host app <-> generated app boundary is UNTRUSTED. Tools are
       adversarial. The Runtime SDK surface is fixed at host build time.
    2. The client <-> LLM provider boundary is INDIRECT. The provider API
       key is server-side only (REQ-SEC-001). The browser authenticates to
       our backend; our backend authenticates to the provider.

10.6  WHAT v1.1 IS NOT (so a reviewer cannot accuse us of over-claiming)
--------------------------------------------------------------------------------

  - v1.1 is NOT iOS-compliant. We do not ship an iOS build. Doc 07 §7.6
    items are tracked in COMPLIANCE.md; most are `deferred` or `n/a-web`.
  - v1.1 is NOT Android-compliant for the same reason.
  - v1.1 does NOT run Tier-2 agentic builds. Hard prompts that Tier-1
    cannot deliver are surfaced honestly (REQ-GEN-008), not built at all
    in v1.1.
  - v1.1 does NOT sync across devices. A tool made on phone A lives on
    phone A. An account is still useful for the server-side quota and
    entitlements; cross-device library sync is v1.0+ (REQ-DATA-002).
  - v1.1 does NOT have a marketplace, gallery, social layer, plug-in
    system, or any "share the tool with another user" path. Per Doc 01
    §7 these are out of scope at any version.
  - v1.1 does NOT have a moderation queue or input/output safety screen.
    The launch is a creator-facing, low-traffic web app; a pre-launch
    human review of the prompt corpus is the v1.1 safety posture, and
    CMP-12 is ROADMAP.md.
  - v1.1 does NOT have a published Privacy Policy / Terms of Service.
    This is the v1.1 launch blocker (OQ-02; COMPLIANCE.md).

10.7  WHEN TO UPDATE THIS DOCUMENT
--------------------------------------------------------------------------------

  - When a v1.0 requirement changes status in v1.1 (a new shipped item,
    a new partial, a deferred item that is now back in scope because the
    v1.0 work has begun). Update §10.4 and §10.5.
  - When v1.0+ work begins in earnest (i.e. when we open the iOS or
    Android workstream). Add a v1.0+ section that re-asserts the
    deferred rows, mark this overlay as historical.
  - When the bible (Docs 00-09) re-baselines. The overlay's role is to
    absorb the delta, not to fight the bible. Edit in place; do not
    delete — historical overlays are useful for understanding why the
    scope changed.
  - For day-to-day status (which requirement is shipped this minute,
    which eval passed this run, which OQ is blocked on what), edit
    STATUS.md. This overlay is the static framing; STATUS.md is the
    live state.
