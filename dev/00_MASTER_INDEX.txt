0.1  HOW TO READ THIS BIBLE
--------------------------------------------------------------------------------

This is a build-to-spec package modeled on the NASA systems-engineering "V"
framework. The left side of the V defines and decomposes the system (vision ->
requirements -> architecture -> components). The bottom is implementation. The
right side verifies and validates each level back up against the specs that
created it. Traceability runs through the whole thing: every test exists to
prove a requirement, and every requirement exists to serve the vision.

You do not need to read all ten documents to start. Read in this order:

   1. Doc 01 (Vision / ConOps)      — what we are building and why. 20 min.
   2. Doc 02 (Requirements)         — the contract. What "done" means.
   3. Doc 03 (Architecture)         — how the system is shaped.
   4. Doc 05 (Implementation Plan)  — sequence, estimates, team, timeline.

Then 04 (interfaces), 06 (V&V), 07 (store approval), 08 (business), 09 (risk)
as each becomes relevant to the phase you are in.

0.5  GLOSSARY
--------------------------------------------------------------------------------

  App (Wild Card "app")  A user-owned mini-tool: a self-contained HTML/CSS/JS
                         bundle plus metadata (name, icon, version). NOT a
                         native binary. Runs inside a WebView on the device.

  Generation            The act of turning a natural-language prompt into a
                        working app bundle. Happens on the server. Two tiers
                        (single-call and agentic).

  Execution / Runtime   Running a finished app on the user's phone, inside a
                        WebView (WebKit on iOS, WebView/Chromium on Android).
                        No server involved at run time.

  Tier-1 generation     One strong model call plus a bounded self-repair loop.
                        Covers the large majority of requests. Fast, cheap.

  Tier-2 generation     A server-side agentic build (Claude Agent SDK in a
                        cloud sandbox) that iterates until tests pass. For hard
                        or multi-file apps. Slower, metered.

  Bridge / Capability   A controlled channel that lets app code request a device
                        capability (save a file, read location). On iOS this is
                        DELIBERATELY MINIMAL for App Store compliance (see Doc 07);
                        on Android it can be richer.

  Runtime SDK           A small JavaScript library injected into every app so it
                        has a consistent, documented surface (storage, UI helpers,
                        a way to surface output to the host). See Doc 04.

  Eval harness          Automated checks that load a freshly generated app in a
                        headless browser and confirm it runs before it reaches
                        the user. The single most important quality system.

  ConOps                Concept of Operations (Doc 01). The human-level story of
                        what the system does and for whom.

  V&V                   Verification (did we build it right?) and Validation
                        (did we build the right thing?). Doc 06.

  MRR / COGS / ARPU     Monthly recurring revenue / cost of goods sold (here,
                        mostly LLM tokens + store fees) / average revenue per user.

0.6  DOCUMENT CONTROL
--------------------------------------------------------------------------------

  Change control:  Any change to a baselined requirement (Doc 02) requires a
                   one-line entry in the change log of that document and a note
                   of which test cases and components are affected. Architecture
                   changes that alter an interface (Doc 04) require sign-off from
                   the mobile lead and backend lead.

  Source of truth: When documents disagree, precedence is:
                   Doc 02 (Requirements) > Doc 03 (Architecture) > everything else.
                   The Vision (Doc 01) governs intent but is not itself testable.

  Estimates note:  All line-of-code (LOC) and time numbers in this bible are
                   engineering estimates with stated assumptions, not commitments.
                   They exist for planning and staffing. Re-baseline after Phase 0.
